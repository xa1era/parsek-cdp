# parsek-cdp

**Асинхронный тулкит для управления Chromium в масштабе, поверх
[Chrome DevTools Protocol](https://chromedevtools.github.io/devtools-protocol/).**

Цель — *удалённое, масштабное* управление браузером: парк экземпляров Chromium,
запускаемых и надзираемых на серверных воркерах, каждым из которых управляешь так,
будто он локальный. Вы пишете против типизированного API, выглядящего как
локальный; тот же код без изменений работает с браузером на другой машине — за
прокси, который надзирает за ним, агрегирует поток событий и отдаёт наружу для
эксплуатации.

Что делает это практичным:

- **Идентичный локальный и удалённый API.** `Browser.connect_http(...)`
  (локально) и `Browser.get_distant_browser(...)` (удалённо, через
  `parsek-cdp-server`) возвращают один и тот же объект с одними и теми же
  методами — разрабатывайте на локальном Chrome, разворачивайте на парке без
  изменений кода.
- **Один websocket на target, без мультиплексирования по `sessionId`.** У каждого
  браузера, страницы, worker'а и OOPIF — своё соединение. Прокси связывает
  клиента с Chrome 1:1, поэтому маршрутизация проста и горизонтально
  масштабируема — stateless-шлюз может разводить клиентов по воркер-подам чисто по
  id браузера.
- **Серверная агрегация событий.** Продюсеры фич сворачивают пачку сырого CDP
  (например, весь `Network.*`) в пару событий `Parsek.*` и подавляют сырые, так
  что высоконагруженная страница не затапливает канал между воркером и клиентом.
- **Жизненный цикл, метрики и очистка из коробки.** Детект падения и рестарт,
  закрытие по простою, метрики Prometheus и отдельный reaper для утёкших
  процессов браузера — всё, что нужно, чтобы крутить сотни браузеров без
  присмотра.

Слой протокола **полностью генерируется** из официальных JSON-описаний
[devtools-protocol](https://github.com/ChromeDevTools/devtools-protocol) — по
одному типизированному модулю на домен. Рантайм написан руками. Требуется
**Python 3.13+** (API использует дженерики PEP 695 с дефолтами type-параметров PEP 696).

> Documentation in English: [README.md](README.md).

## Пакеты

Репозиторий — монорепа из двух отдельно публикуемых дистрибутивов:

| Пакет | Установка | Зона ответственности |
| --- | --- | --- |
| [`parsek-cdp`](packages/client) | `pip install parsek-cdp` | Клиент: сгенерированный слой протокола `cdp`, рантайм `core` (browser / page / frame / element), система фич и контракт `parsek`. |
| [`parsek-cdp-server`](packages/server) | `pip install parsek-cdp[server]` | Сервер: запуск, надзор и проксирование браузера; хостинг продюсеров фич; метрики Prometheus; reaper зомби-процессов. Зависит от `parsek-cdp` и переиспользует его общие слои. |

Клиент никогда не зависит от сервера — связывает их только extra `[server]`.
Полное использование — в README каждого пакета:
[клиент](packages/client/README.ru.md) · [сервер](packages/server/README.ru.md).

## Быстрый старт

Управление локальным браузером (запущенным как `chromium --headless=new --remote-debugging-port=9222`):

```python
import asyncio
from parsek_cdp import Browser, Page, RequestListener


async def main():
    browser = await Browser.connect_http(
        "http://127.0.0.1:9222", page_class=Page[RequestListener]
    )
    page = await browser.new_page()
    requests = page.get_feature(RequestListener)

    await page.navigate("https://example.com", wait_load=True)
    for r in requests.requests:
        print(r.method, r.url, "->", r.response.status_code if r.response else "(ещё нет ответа)")

    await browser.close()


asyncio.run(main())
```

Тот же код управляет *удалённым* браузером через `parsek-cdp-server` — меняется
только точка входа. Сервер запускает и надзирает за браузером; объявленные вами
фичи агрегируются на стороне сервера:

```python
browser = await Browser[Page[RequestListener]].get_distant_browser(
    "http://127.0.0.1:9333", headless=True
)
```

## Абстракции

Клиентский рантайм (`core/`) — небольшой набор композируемых объектов. Каждый
слой добавляет ровно одну ответственность поверх предыдущего.

### `CDPConnection` — один websocket, JSON-RPC

Основа всего: один websocket, несущий одну DevTools-сессию. Владеет циклом
приёма, назначает id командам и сопоставляет каждый ответ с ожидающим future, а
также диспетчеризует приходящие события. Намеренно изолирует сбои — кривой кадр
или упавший обработчик события логируются и пропускаются, не роняя сокет, — а
когда соединение всё-таки закрывается, все ожидающие future команд завершаются
`ConnectionError`, а не висят вечно.

### `Target` — CDP-сессия со своими детьми

`Target` — это и есть `CDPConnection` (браузер, страница, worker, OOPIF), так что
у каждого target'а свой websocket — `{host}/devtools/{type}/{id}`. Хост
привязывается один раз в `contextvar` при подключении к браузеру и наследуется
каждым target'ом, подключённым изнутри. Поверх сырого соединения target добавляет:

- **всю командную поверхность** через `target.cdp.<Domain>` — по атрибуту на
  каждый CDP-домен, с типизированными методами и результатами;
- **учёт доменов** — каждый `enable`/`disable` отслеживается по сессии (и
  процессно), так что `domain_enabled(...)` включает домен на блок и
  восстанавливает прежнее состояние, не включая дважды;
- **обработчики событий** — `on` / `off` на target'е, плюс `wait_for(Event)`,
  чтобы приостановиться до первого срабатывания; обработчики могут быть синхронными
  и асинхронными;
- **хуки на команды** — `on_function(cmd, handler)` срабатывает каждый раз, когда
  данная CDP-команда *отправляется* (двойник обработчика события);
- **живое дерево дочерних target'ов** — `refresh_targets()` перестраивает поддерево
  с корнем в этом target'е по авторитетному снимку `Target.getTargets` (iframe'ы
  вложены под родительский фрейм, popup'ы — под opener'а), переиспользуя
  существующие объекты `Target`, так что их идентичность и открытые сокеты
  переживают обновление.

```python
await page.cdp.Network.enable()
info = await page.cdp.Target.get_target_info()

async with page.domain_enabled(page.cdp.Page):       # включить на блок, потом восстановить
    await page.cdp.Page.navigate(url="https://example.com")

from parsek_cdp.cdp import Network
page.on(Network.RequestWillBeSent, lambda e: print(e.request.url))   # на target'е

@Network.RequestWillBeSent.add_handler                               # глобально, на любом target'е
async def on_request(e):
    print(e.request.url)

evt = await page.wait_for(Network.ResponseReceived, timeout=10)      # дождаться одного события
```

### `Browser` — эндпоинт и обнаружение target'ов

`Browser[T]` подключается к browser-level websocket'у, включает обнаружение
target'ов и отдаёт target'ы страниц как объекты `Page` (типизированные по `T` —
классу страницы, который вы передаёте). Он дженерик по этому классу, так что
`Browser[Page[RequestListener]]` протягивает набор фич в `browser.pages`,
`new_page()` и контексты. Два входа:

- `Browser.connect_http(endpoint)` — обнаружить websocket через `/json/version` и
  подключиться к **локальному** Chrome;
- `Browser.get_distant_browser(server, ...)` — `POST /browsers` на
  `parsek-cdp-server`, затем подключение к control-каналу прокси; возвращённый
  браузер управляет **удалённым** ровно как локальным, с фичами, агрегируемыми на
  сервере (страницы исполняют фичи как *view*, а не producer'ы).

### `BrowserContext` — изолированные профили

Страницы группируются в контексты браузера (профили в духе инкогнито,
опционально со своим прокси). Страница, зарегистрированная в контексте,
«всплывает» в глобальный реестр браузера — один экземпляр `Page`, общий для обоих,
— так что можно работать и по контексту, и по всему браузеру.

```python
browser = await Browser.connect_http("http://127.0.0.1:9222")
context = await browser.create_context(proxy_server="http://10.0.0.1:8080")
page = await context.new_page("https://example.com")
print(len(browser.pages), "страниц в", len(browser.contexts), "контекстах")
```

### `Page`, `Frame`, `Element` — слой, обращённый к DOM

`Page` — это `Target`, `FeatureHost` **и** `Frame`: в CDP id главного фрейма равен
id target'а страницы, поэтому страница *сама является своим главным фреймом*, и
`page.select(...)`, `page.evaluate(...)`, `page.url` работают по верхнему фрейму
без отдельного объекта. Полное дерево фреймов — вложенные и кросс-доменные
**OOPIF**, каждый прозрачно маршрутизируемый через свою сессию — живёт в
`page.frames`.

- **`Frame`** ограничивает область запросов и исполнения: `select` / `select_all`
  (CSS *или* XPath), `evaluate`, `wait_for_selector`, `wait_for_xpath`. Каждый
  запрос исполняется в документе этого фрейма, так что подфреймы резолвятся верно.
- **`Element`** — живой хэндл DOM-узла: `text`, `attributes`, `children`,
  `query_selector(_all)`, плюс действия — `mouse_click`, `fill`, `type`, `press`,
  `focus`, `scroll_into_view`, `is_visible` и `apply(jsFn)` для запуска
  JS-функции с узлом в роли `this`.
- **помощники навигации** у страницы: `navigate(url, wait_load=True)` взводит
  ожидание загрузки *до* навигации (так быстрый load не теряется), а
  `with_same_navigation()` отменяет блок в момент, когда главный фрейм
  фиксирует новый документ — чтобы логика не продолжала работать по странице,
  ушедшей из-под неё.

```python
el = await page.select(selector="input[name=q]")
await el.fill("parsek")
await el.press("Enter")
print(el.text, el.attributes)

title = await page.evaluate("document.title")

async with page.with_same_navigation():          # прервётся, если страница навигирует
    row = await page.select(selector="#row")
    await row.mouse_click()
```

### `Feature` / `FeatureHost` — опциональное поведение: пишется раз, работает везде

`Page` по умолчанию не подключает **никакого** поведения; *фича* — это единица
поведения, которую вы подключаете: `Page[F]` (декларативно, типизировано) или
`page.get_feature(F)` (императивно). Смысл системы фич в том, что одна реализация
работает в **трёх ролях** без изменений:

- **local** — напрямую к браузеру: продюсеры фичи подписываются на сырой CDP на
  странице и редьюсят его в процессе (обычная локальная `Page`);
- **server** — прокси скармливает сырой CDP продюсеру, чей вывод *эмитится*
  клиенту как событие `Parsek.*` **и** редьюсится локально для снимков;
- **client** — выполняются только редьюсеры, питаемые этими событиями `Parsek.*`
  с провода.

Поскольку редьюсеры выполняются в каждой роли, публичный API фичи (состояние,
которое она строит) одинаков и для локального, и для удалённого браузера — именно
это и позволяет масштабировать тот же код. `FeatureHost` — миксин, дающий
странице композицию `[F]` и `attach` / `get_feature` / `has_feature`, откладывая
асинхронный `start()` каждой фичи до момента, когда соединение поднято.
`RequestListener` — встроенная эталонная фича; своя пишется по «доменной»
раскладке пакета (`types` · `events` · класс `Feature`) — полный гайд в
[README клиента](packages/client/README.ru.md#фичи).

### Контракт `parsek`

Маленький срез протокола, который *не* является обычным CDP — жизненный цикл
браузера под надзором сервера (`BrowserState`, `BrowserStateChanged`). Поставляется
в клиентском дистрибутиве (`parsek_cdp.parsek`), чтобы обе стороны делили одно
определение провода.

## Серверные абстракции

`parsek-cdp-server` добавляет то, что делает удалённые браузеры эксплуатируемыми;
эндпоинты и метрики — в его [README](packages/server/README.ru.md).

- **`ChromeLauncher`** — находит Chromium-совместимый бинарь (или случайный среди
  нескольких установленных браузеров), запускает его, ждёт DevTools-порт и
  считывает websocket. Владеет временным каталогом профиля и убирает его.
- **`BrowserSupervisor`** — владеет одним браузером как «задачей»: опрос здоровья,
  детект падения и рестарт, закрытие по простою и broadcast каждого перехода
  состояния, чтобы клиент реагировал, а не висел на мёртвом сокете.
- **`ParsekServer` / `PageBridge`** — HTTP/ws-фронт. Control-канал пробрасывает
  browser-эндпоинт плюс `Parsek.browserStateChanged`; каждый page-pipe — это
  `PageBridge`, который ретранслирует сырой CDP 1:1 и хостит продюсеров фич,
  подавляя сырые события, которые они переварили (`?raw=1` отключает подавление).
- **`ServerMetrics`** — экспозиция Prometheus (браузеры, targets по типам,
  вложенные targets, счётчики CDP-событий по target'у, CPU/RAM дерева процессов),
  снимаемая вне event loop.
- **reaper** — отдельный субпроцесс, убивающий утёкшие (зомби/осиротевшие)
  Chrome-процессы, запущенные нами, опознаваемые по маркеру каталога профиля.

Предполагаемая топология в масштабе: **stateless-шлюз** маршрутизирует каждый
клиентский websocket к воркеру, владеющему браузером, чисто по id браузера (без
общего хранилища), а каждый воркер крутит `ParsekServer`, надзирающий за своими
браузерами.

## Кодогенерация

Пакет `cdp/` написан **не** руками — он генерируется из апстрим-JSON протокола
скриптом `parsek_cdp/generate.py`. На каждый домен выпускается:

| Модуль | Содержимое |
| --- | --- |
| `types.py` | enum'ы (`str, Enum`), object-типы (`@dataclass` поверх `DataType`) и алиасы (PEP 695 `type X = ...`) |
| `functions.py` | каждая команда — типизированный `async`-метод на классе домена, привязанном к target'у; возвращает сгенерированный датакласс `…Return` |
| `events.py` | каждое событие — датакласс `Event`, зарегистрированный для диспетчеризации обработчиков |
| `__init__.py` | пространство имён домена; верхнеуровневый `cdp/__init__.py` собирает все домены в аксессор `CDP` за `target.cdp` |

Почему нет циклов импортов: каждый `$ref` разрешается **на этапе генерации**
(цепочки алиасов разворачиваются до конкретной формы), поэтому в рантайме нужно
лишь лениво находить enum'ы и object-типы по полному имени. Кросс-доменные ссылки
на типы выводятся ленивыми строковыми аннотациями, импортируемыми только под
`TYPE_CHECKING`, так что присущие протоколу циклические кросс-доменные импорты
никогда не материализуются в рантайме. Каждый сгенерированный тип несёт таблицу
`__FIELDS__` из `FieldMeta` — она описывает соответствие `snake_case` ↔
`camelCase` и то, как (де)сериализуются вложенные значения (enum/объект/массив).

### Регенерация / свои JSON протокола

Генератор принимает один аргумент — commit/tag/branch devtools-protocol
(скачивается с GitHub) либо путь к локальному каталогу с `js_protocol.json` +
`browser_protocol.json`:

```bash
# зафиксировать апстрим-ревизию (commit, tag или branch) и скачать её
python -m parsek_cdp.generate master

# либо сгенерировать из локальной копии JSON протокола
python -m parsek_cdp.generate path/to/protocol-json/
```

Результат пишется в `packages/client/parsek_cdp/cdp/`. Так нацеливаются на
конкретную версию Chrome или на форк/расширение протокола: укажите генератору
соответствующий JSON — и вся типизированная поверхность пересоберётся.
